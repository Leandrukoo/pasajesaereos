1. Iniciar Sesión index.html → scripts/index.js
El formulario de login verifica que el email y la contraseña ingresados coincidan con el usuario guardado en localStorage. Si no hay usuario registrado, se compara contra credenciales de prueba por defecto.
Campos del formulario
Campo Tipo HTML Regla ¿Cuándo valida? Mensaje de error
#email type="email" REQUERIDO HTML5 Debe coincidir con el email registrado AL ENVIAR "Email o contraseña incorrectos"
#password type="password" REQUERIDO HTML5 Debe coincidir con la contraseña registrada AL ENVIAR "Email o contraseña incorrectos"
Lógica de autenticación

Al hacer submit, manejarLogin() sigue este flujo:

1. Lee email + password del formulario
→
2. Lee usuarioRegistrado de localStorage
→
3. Compara credenciales
→
4. Éxito / Error
Si las credenciales coinciden:
Guarda el objeto del usuario en localStorage bajo la clave usuarioLogueado. Redirige a pages/inicio.html.
Si no coinciden:
Muestra una notificación de error: "Email o contraseña incorrectos". El usuario permanece en la página de login.
Credenciales de prueba (fallback): Si no hay ningún usuario registrado en localStorage, se compara contra email test@test.com y contraseña 1234.
Feedback visual

Los campos usan required de HTML5: el navegador bloquea el envío y muestra un tooltip nativo si algún campo está vacío o el email no tiene formato válido.
Los errores de autenticación (credenciales incorrectas) se muestran mediante mostrarNotificacion() de utils.js — un bloque rojo flotante en la esquina superior derecha que desaparece a los 3 segundos.

2. Registro de usuario pages/registrarse.html → scripts/registrarse.js
El formulario de registro crea una cuenta nueva. Valida que los campos obligatorios estén completos, que las contraseñas coincidan y que tengan la longitud mínima requerida. Al registrarse exitosamente, guarda el usuario en localStorage.
Campos del formulario
Campo Tipo HTML Regla ¿Cuándo valida? Mensaje de error
Nombre type="text" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Email type="email" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
#password type="password" REQUERIDO MÍN 4 CHARS Contraseña principal AL ENVIAR "La contraseña debe tener al menos 4 caracteres"
#confirm-password type="password" REQUERIDO Debe ser idéntico a #password AL ENVIAR "Las contraseñas no coinciden"
Lógica de registro

Al hacer submit, manejarRegistro() sigue este flujo:

1. HTML5 verifica campos required
→
2. Compara password vs confirm-password
→
3. Verifica longitud mínima (4)
→
4. Guarda en localStorage
→
5. Redirige a login
Si todo es válido:
Guarda el usuario en localStorage con clave usuarioRegistrado (objeto con nombre, email y password). Redirige al usuario a index.html para que inicie sesión.
Si hay error:
Muestra notificación con el mensaje de error correspondiente. No guarda ni redirige.
Feedback visual

Los campos vacíos y el email con formato inválido son capturados por HTML5 (tooltip nativo del navegador).
Los errores de contraseña se muestran con mostrarNotificacion() — notificación flotante roja. No hay coloreado de bordes campo por campo en esta página.

3. Búsqueda de vuelos pages/inicio.html → scripts/inicio.js
El formulario de búsqueda de vuelos valida el tipo de viaje, las fechas y la coherencia entre ellas. También deshabilita el campo de fecha de vuelta cuando el tipo es "solo ida".
Campos del formulario
Campo Tipo HTML Regla ¿Cuándo valida? Mensaje de error
Tipo de viaje (radio) type="radio" REQUERIDO Al menos una opción debe estar seleccionada AL ENVIAR "Por favor seleccioná el tipo de viaje"
Origen type="text" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Destino type="text" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Fecha de ida type="date" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Fecha de vuelta type="date" OPCIONAL RANGO No puede ser anterior a la fecha de ida. Deshabilitado en modo "solo ida". AL CAMBIAR AL ENVIAR "La fecha de vuelta no puede ser anterior a la de ida"
Pasajeros type="number" REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Clase (select) <select> REQUERIDO HTML5 AL ENVIAR Tooltip nativo del navegador
Comportamiento especial: tipo de viaje

manejarCambioTipo() escucha el cambio en los radio buttons de tipo de viaje y actualiza el estado del campo de fecha de vuelta:

Ida y vuelta seleccionado:
Habilita el campo #fecha-vuelta (quita el atributo disabled).
Solo ida seleccionado:
Deshabilita el campo #fecha-vuelta y limpia su valor. El campo queda visualmente bloqueado.
Lógica al enviar

Al hacer submit, manejarEnvio() sigue este flujo:

1. Verifica que un radio esté seleccionado
→
2. HTML5 verifica required
→
3. Si ida-vuelta: valida rango de fechas
→
4. Guarda búsqueda en localStorage
→
5. Redirige a vuelos.html
Persistencia: los datos de la búsqueda (origen, destino, fechas, pasajeros, clase) se guardan en localStorage bajo la clave busquedaVuelo para que vuelos.html los muestre. Al cargar la página, cargarDatosFormulario() restaura los valores guardados.
Feedback visual

Los errores de campos vacíos se muestran con los tooltips nativos del navegador (atributo required).
El error de rango de fechas (vuelta anterior a ida) se muestra con mostrarNotificacion() — notificación flotante roja — y limpia el valor del campo de fecha de vuelta.
El campo de fecha de vuelta deshabilitado queda visualmente atenuado por los estilos del navegador.

4. Datos del pasajero y pago pages/checkout.html → scripts/checkout.js
La página de checkout es la más completa en cuanto a validaciones. Cada campo tiene su propia función validadora que se ejecuta tanto al perder el foco (blur) como al intentar enviar el formulario. Incluye además un sistema de cupones de descuento.
Campos del formulario
Campo Función validadora Regla ¿Cuándo valida? Mensaje de error
#nombre-pasajero validarNombre() REQUERIDO MÍN 3 CHARS AL PERDER FOCO AL ENVIAR "El nombre debe tener al menos 3 caracteres"
#dni-pasajero validarDNI() REQUERIDO 7–10 DÍGITOS Solo números (se ignoran guiones/espacios) AL PERDER FOCO AL ENVIAR "Ingresá un DNI válido (7 a 10 dígitos)"
#email-pasajero validarEmail_checkout() REQUERIDO REGEX Formato: usuario@dominio.ext AL PERDER FOCO AL ENVIAR "Ingresá un email válido"
#telefono-pasajero validarTelefono_checkout() OPCIONAL 10–15 DÍGITOS Solo se valida si tiene contenido AL PERDER FOCO AL ENVIAR "Ingresá un teléfono válido (10 a 15 dígitos)"
Método de pago (radio) — REQUERIDO Al menos una opción debe estar seleccionada AL ENVIAR "Seleccioná un método de pago"
Feedback visual por campo

Cada función validadora aplica estilos directamente al campo mediante aplicarEstiloValidacion(elemento, esValido):

Campo válido:
Borde: 2px solid #4caf50 (verde)
Sombra: box-shadow: 0 0 5px rgba(76,175,80,0.3)
Mensaje de error: oculto
Campo inválido:
Borde: 2px solid #f44336 (rojo)
Sombra: box-shadow: 0 0 5px rgba(244,67,54,0.3)
Mensaje de error: visible debajo del campo

El feedback se actualiza en tiempo real: al perder el foco el campo se colorea, y si el usuario corrige el valor y vuelve a hacer blur, el color cambia inmediatamente. Al enviar, todos los campos se revalidan de una vez.

Sistema de cupones de descuento

aplicarCupon() verifica el código ingresado en #cupon contra una lista de cupones válidos (case-insensitive — convierte a mayúsculas antes de comparar):

Código Descuento Efecto
DESCUENTO10 10% Multiplica el precio base por 0.90
DESCUENTO20 20% Multiplica el precio base por 0.80
VIAJERO 15% Multiplica el precio base por 0.85
PROMO2024 5% Multiplica el precio base por 0.95
Cupón válido:
Actualiza el precio mostrado con el descuento aplicado. Muestra notificación verde: "Cupón aplicado: X% de descuento".
Cupón inválido:
Muestra notificación roja: "Cupón no válido". El precio no cambia.
Lógica al enviar

Al hacer submit, manejarEnvioCheckout() sigue este flujo:

1. Llama a las 4 funciones validadoras
→
2. Verifica que haya método de pago seleccionado
→
3. Si alguno falla → detiene el envío
→
4. Si todo OK → muestra confirmación
Nota: Si algún campo falla, el formulario no se envía y el scroll lleva automáticamente al primer campo con error para que el usuario lo corrija.
5. Formulario de contacto pages/contacto.html → scripts/contacto.js
El formulario de contacto combina validación de campos con autosave automático. Los datos se guardan en localStorage mientras el usuario escribe, de forma que no se pierdan si cierra la pestaña accidentalmente.
Campos del formulario
Campo Función validadora Regla ¿Cuándo valida? Mensaje de error
#nombre-contacto validarNombreContacto() REQUERIDO MÍN 3 CHARS AL PERDER FOCO AL ENVIAR "El nombre debe tener al menos 3 caracteres"
#email-contacto validarEmailContacto() REQUERIDO REGEX Formato: usuario@dominio.ext AL PERDER FOCO AL ENVIAR "Ingresá un email válido"
#telefono-contacto validarTelefonoContacto() OPCIONAL 10–15 DÍGITOS Solo se valida si tiene contenido AL PERDER FOCO AL ENVIAR "Ingresá un teléfono válido"
#mensaje-contacto validarMensajeContacto() REQUERIDO MÍN 10 CHARS AL PERDER FOCO AL ENVIAR "El mensaje debe tener al menos 10 caracteres"
Feedback visual por campo

Todas las funciones validadoras usan aplicarEstiloValidacion(elemento, esValido), que aplica el mismo esquema de colores que en checkout:

Campo válido:
Borde: 2px solid #4caf50 (verde)
Sombra: 0 0 5px rgba(76,175,80,0.3)
Mensaje de error: oculto
Campo inválido:
Borde: 2px solid #f44336 (rojo)
Sombra: 0 0 5px rgba(244,67,54,0.3)
Mensaje de error: visible debajo del campo
Autosave automático

guardarDatosFormularioContacto() escucha el evento input en cada campo. Cada vez que el usuario escribe un carácter, guarda todos los datos del formulario en localStorage bajo la clave formularioContacto.

Al cargar la página:
inicializarFormularioContacto() lee formularioContacto de localStorage y rellena los campos automáticamente si hay datos guardados. El usuario nunca pierde lo que ya había escrito.
Al hacer click en "Limpiar":
Se muestra un diálogo de confirmación: "¿Estás seguro de que querés borrar el formulario?". Si el usuario confirma, limpia los campos Y borra los datos de localStorage.
Datos guardados: nombre, email, teléfono, asunto y mensaje. El guardado ocurre en tiempo real con cada pulsación de tecla — no hay que esperar a enviar el formulario.
Lógica al enviar

Al hacer submit, manejarEnvioContacto() sigue este flujo:

1. Llama a las 4 funciones validadoras
→
2. Si alguno falla → detiene el envío
→
3. Si todo OK → muestra confirmación
→
4. Limpia formulario y localStorage
Envío exitoso:
Muestra un mensaje de confirmación en la página: "¡Mensaje enviado con éxito!" (función mostrarConfirmacionEnvio()). Limpia el formulario y borra los datos guardados de autosave para no restaurarlos en la próxima visita.
